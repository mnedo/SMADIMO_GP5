# Role C: Two-Branch Residual MLP для предсказания цены товаров Oskelly

## Описание части
В этой части была реализована нейросетевая модель для предсказания цены товара на основе табличных характеристик товара и текстовых признаков, которые были представлены в виде эмбеддингов описания. Целью было построить MLP архитектуру, которая может учитывать данные о товаре и информацию из текста.
На вход модели подается массив x, который состоит из двух частей - первые 384 признака текстовые эмбеддинги, а оставшиеся 204 признака табличные признаки.

## Размерность данных: X: (9268, 588) y: (9268,)
Данные были разделены на обучающую, валидационную и тестовую выборки:
- train: 6487 объектов
- validation: 1390 объектов
- test: 1391 объект

Целевая переменная у нас цена товара. Перед обучением мы ее преобразовали через log1p, а затем стандартизировали по среднему и стандартному отклонению обучающей выборки. После предсказания значения возвращались обратно в исходную шкалу цены.

Для обучения модели выполнялись такие шаги:
1.	Разделение признаков на текстовые и табличные
2.	Стандартизация эмбеддингов с помощью StandardScaler
3.	Приведение всех признаков и целевой переменной к типу float32
4.	Перевод данных в формат torch.tensor
5.	Создание TensorDataset и DataLoader для train, validation и test
6.	random seed = 42.

Также была реализована модель TwoBranchMLP.
Архитектура состоит из двух веток. Первая: Tabular encoder - обрабатывает табличные признаки. Вторая: Text encoder - обрабатывает текстовые эмбеддинги.
Каждая ветка содержит: Linear, BatchNorm1d, ReLU, Dropout, еще один Linear и ReLU.
После этого выходы двух веток объединяются через torch.cat, проходят через общий слой объединения и residual-блок. Residual-блок был добавлен для того, чтобы модель могла дообучать представление признаков, при этом не потеряв исходную информацию после объединения веток.

Для обучения использовались: Loss: MSELoss, Optimizer: AdamW, Scheduler: ReduceLROnPlateau, Early stopping: по validation loss, Batch size: 64

Эксперименты логировались в Comet ML.В лог сохранялись параметры запуска, train loss, validation loss, лучшая эпоха, тестовые метрики и финальная модель.

Было проведено 12 запусков с разными значениями learning rate, dropout, branch_dim, hidden_dim, patience, функции потерь MSELoss и HuberLoss.

## Transfer Learning - EfficientNet-B3

## Выбор архитектуры

Выбрана **EfficientNet-B3** по следующим причинам:

- Использует compound scaling - одновременное масштабирование глубины, ширины и разрешения сети, что даёт лучшее соотношение качество/размер модели по сравнению с ResNet
- Одна из лучших известных моделей; на ImageNet выдает показатели лучше, чем, например, ResNet
- Среди всех EfficientNet-B{X} не слишком слабая и не слишком мощная (для приемлевого срока обучения)
- Архитектура не пересекается с CLIP-ViT (трансформер), который используется в соседнем эксперименте

Стандартные mean/std нормализации соответствуют ImageNet: `[0.485, 0.456, 0.406]` / `[0.229, 0.224, 0.225]`. ([Отсюда взяли значения](https://docs.pytorch.org/vision/stable/models/generated/torchvision.models.efficientnet_b3.html))

Для train применены лёгкие аугментации, агрессивные аугментации не использовались, так как у нас относительно небольшой датасет. Сильные трансформации могут значительно ухудшить качество. Val-трансформы включают только Resize и Normalize.

Выбрана трехэтапная стратегия постепенной разморозки backbone. При полной заморозке backbone модель не может адаптировать низкоуровневые признаки под специфику наших данных, а при полной разморозке сразу имеется риск переобучения на сравнительно малом датасете. Поэтапная разморозка позволяет сначала обучить голову, а затем потихоньку дообучить backbone.

### Stage 1 - Feature Extraction (frozen backbone)

- Заморожены все слои backbone, обучается только классификатор
- Классификатор: `Dropout(0.3) → Linear(1536, 13)`
- Optimizer: Adam, lr=1e-3
- Epochs: 10
- Логирование: [Comet ML - EfficientNetB3 stage1](https://www.comet.com/gp5-team1/oskelly-cv/3202da114b8741d797e2ae4efc07294e)

На последней эпохе (десятой) было достигнуто значение 0.7024 для метрики f1.

Вывод: backbone хорошо извлекает признаки одежды даже без дообучения - val_f1 вырос с 0.54 до 0.70 за 10 эпох только за счет обучения классификатора.

### Stage 2 - Fine-tuning (разморозка features.7–8)

- Разморожены последние два блока backbone (`features.7`, `features.8`) + классификатор
- Optimizer: Adam, lr=1e-4 (в 10 раз меньше, чтобы не разрушить предобученные веса)
- Epochs: 5
- Логирование: [Comet ML - EfficientNetB3 stage2](https://www.comet.com/gp5-team1/oskelly-cv/20f157c0e6be4bac8398a89d8489f9fe)

На четвертой эпохе (из пяти) было достигнуто значение 0.7672 для метрики f1.

Вывод: разморозка последних блоков дала существенный прирост: val_f1 поднялся с 0.70 до почти 0.77. Блоки `features.7–8` отвечают за высокоуровневые признаки (силуэт, форма), которые наиболее релевантны для fashion-классификации.

### Stage 3 - Deep Fine-tuning (разморозка features.5–8)

- Разморожены блоки `features.5–8` + классификатор
- Optimizer: Adam, lr=1e-5 (еще в 10 раз меньше)
- Epochs: 5
- Логирование: [Comet ML - EfficientNetB3 stage3](https://www.comet.com/gp5-team1/oskelly-cv/3330f2fb5e9f499886b6e0e60d31ed45)

На четвертой эпохе (из пяти) было достигнуто значение 0.7679 для метрики f1.

Вывод: дальнейшая разморозка дала минимальный прирост (менее 0.001 f1), а на последней эпохе val_loss резко вырос - признак начала переобучения. Значит, обучение было остановлено своевременно.

Джинсы (0.92), Майки и футболки (0.92), Шорты (0.90), Брюки (0.90) оказались лучшими классами по метрике f1 - визуально четкие и хорошо представленные классы. Топы с длинным рукавом (0.49), Блузы и рубашки (0.66), Жакеты и пиджаки (0.60), Жилеты (0.60) - наоборот - самые слабые классы, их легко перепутать с другими классами, границы между которыми субъективны. Также низкое качество на этих классах обусловлено их редкостью.

**CLIP** — это модель OpenAI, которую обучали на сотнях миллионов пар (фото + подпись) методом контрастивного обучения: энкодер картинок и энкодер текста притягивались, если пара правильная, и отталкивались, если нет. В итоге image encoder научился выдавать эмбеддинги, которые реально отражают семантику изображения — не просто текстуры и края, как у CNN, а что-то вроде «понимания» что на фото.
Мы этот энкодер замораживаем (не трогаем) и просто вешаем сверху маленький классификатор-голову, который учим уже на наших метках состояния. То есть мы не обучаем 400M параметров — мы обучаем ~1-2 слоя поверх готовых хороших признаков.
